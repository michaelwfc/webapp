Chapter 01 / Sessions

Identifying users across requests

HTTP is stateless. Learn how web servers track who is talking to them using cookies, session IDs, and modern storage APIs.

The statelessness problem

HTTP requests arrive at the server with almost no information to identify which user — or which "session" — they belong to. But web apps need to know who is making the request on every single call.

Browser
→ HTTP →
Web Server (stateless)
→ LAN →
Storage

The web server is inherently stateless — session data must live in storage, with a reference carried in the HTTP request.

Cookies: the classic solution

The server adds a Set-Cookie header to the response. The browser attaches it automatically to every future request to the same domain.

Server sets a cookie
// HTTP Response headers
Set-Cookie: session=0x4137fd6a; Expires=Wed, 09 Jun 2012 10:18:14 GMT

// Browser sends it back automatically
Cookie: session=0x4137fd6a
Security note: Never store sensitive data (like passwords) directly in a cookie. Users can view, modify, and steal cookies. Use a session ID that points to server-side data, protected with cryptographic signing.

Cookie limitations

  • Size limit: typically under 4 KB per cookie
  • Count limit: ~50 cookies per server domain
  • User control: users can view, modify, delete, or block cookies entirely
  • Browser-specific: switching browsers looks like a new user
  • Security risk: cookies can be stolen via XSS attacks

Best practice: use cookies only for hints or session tokens — never as reliable storage.

ExpressJS session middleware

Express provides express-session which handles all the boilerplate: signing the session ID, storing session data, and attaching it to every request.

Setup & usage
const session = require('express-session');

// Mount middleware
app.use(session({ secret: 'yourSecret' }));

// In a route handler
app.post('/login', (req, res) => {
  req.session.userId = 'mendel';  // store on login
  res.redirect('/');
});

app.get('/profile', (req, res) => {
  if (!req.session.userId) {
    return res.redirect('/login');
  }
  // userId is available on every request
});

// On logout
req.session.destroy((err) => { });

Where to store session data

Using MongoDB as session store
const MongoStore = require('connect-mongo')(express);
app.use(session({
  store: new MongoStore({ mongooseConnection: mongoose.connection })
}));

Web Storage API: modern alternative

Modern browsers offer localStorage and sessionStorage — both provide a simple key-value interface with ~10 MB of space, no automatic server transmission.

// sessionStorage — cleared when tab closes
sessionStorage.setItem('theme', 'dark');

// localStorage — persists across sessions
localStorage.username = 'mendel';
localStorage.setItem('token', 'abc123');
Key difference from cookies: Web Storage is never automatically sent to the server. You must read it with JavaScript and include it manually in requests.
Quick check: Why is storing a session ID in a cookie safer than storing the user's actual data in the cookie?
Correct! A session ID is just an opaque reference. Actual session data (user ID, preferences, etc.) stays safely on the server. The ID itself must still be cryptographically signed to prevent forgery — but it reveals nothing sensitive on its own.
Chapter 02 / State Management

Keeping your app in sync

As apps grow from read-only to interactive, coordinating data between components and the server becomes the hardest problem.

Why simple apps are deceptively easy

A read-only photo app with static MVC works fine: each component independently fetches its own data at startup. But the moment you add logins, object creation, or updates, everything gets complicated.

  • A user adds a comment → another component's model is now stale
  • A user logs out → all server requests will start failing
  • Two components fetch the same user → duplicate network requests

Session state: the login/logout problem

The server rejects requests from unauthenticated users, so components that fetch data at startup may fail silently after a logout.

Login transition: most models unavailable → become available after login

Logout transition: requests that worked → now return 401 errors
React Router guarding routes
{
  this.state.userIsLoggedIn
    ? <Route path="/users/:id" component={UserDetail} />
    : <Redirect path="/users/:id" to="/login-register" />
}

Component communication patterns

How do you notify a view component when a separate add component changes the data?

Option A — callbacks: pass functions as props. Works for simple parent-child relationships, gets messy at scale.

<AddPhoto onAdded={this.refreshPhotos.bind(this)} />

Option B — listener/emitter pattern: components subscribe to events they care about; producers emit when data changes. This is what Redux, FLUX, and similar libraries formalize.

Rule of thumb: callbacks are fine for a few components. Once you have 4+ components that share data, invest in a state manager.

Handling external model changes

What if another user adds a photo while you're viewing the list? You have three strategies:

StrategyDescriptionTradeoff
Do nothingNever re-fetch after startupEasy but confusing — user won't see new content
PollPeriodically re-fetch or offer a refresh buttonGood balance; user controls freshness
Server pushServer pushes changes via WebSocketsReal-time but complex; may conflict with edits
A user adds a comment to a photo. The UserPhotos component doesn't show it until the page is refreshed. Which strategy describes this behavior?
Correct! This is the "do nothing" approach — the model is fetched once at startup and never refreshed unless the user manually navigates away and back. It's especially confusing when the user themselves made the change and doesn't see it reflected.
Chapter 03 / Input & Validation

Getting data from users safely

Forms, controlled components, async validation, and why you always need server-side checks regardless of what happens in the browser.

Traditional HTML forms vs React

Classic web apps used native HTML <form> with a full page reload on submit. React replaces this with controlled components — input values are stored in React state, and the DOM is a reflection of that state.

React controlled component pattern
// Input value lives in state, not the DOM
<input
  type="text"
  value={this.state.inputValue}
  onChange={this.handleChange}
/>

handleChange(event) {
  this.setState({ value: event.target.value });
}

handleSubmit(event) {
  // Prevent browser from doing a native POST
  event.preventDefault();
  // Use this.state to process the form
}

The this binding problem

Passing a method as a DOM event callback loses the correct this reference. Three solutions exist — pick one and use it consistently:

// ❌ Wrong — this is undefined in the callback
<form onSubmit={this.formSubmit}>

// ✅ Option 1: Arrow function in JSX
<form onSubmit={e => this.formSubmit(e)}>

// ✅ Option 2: Bind in constructor
this.formSubmit = this.formSubmit.bind(this);

// ✅ Option 3: Class field arrow function
formSubmit = (event) => { /* ... */ };

Validation: two rules that never change

Rule #1: Always validate on the server. It is trivially easy to bypass all browser-side checks by sending HTTP requests directly.

Rule #2: Show the user validation errors as early as possible — ideally on each keystroke, not just on submit.
Custom validation in handleChange
handleChange(event) {
  if (!this.validateIt(event.target.value, this.state)) {
    this.setState({ renderError: true });
  }
  this.setState({ value: event.target.value });
}

React is unopinionated here — you can validate with arbitrary JavaScript or use a library like Formik which handles error reporting, touched state, and schema-based validation.

Asynchronous validation

Some checks require a server round-trip — for example, verifying that a username isn't already taken. This can be done in the background as the user types.

handleUsernameChange(event) {
  const val = event.target.value;
  this.setState({ username: val });

  // Debounced async check
  clearTimeout(this.debounce);
  this.debounce = setTimeout(() => {
    axios.get(`/check-username/${val}`)
      .then(r => this.setState({ usernameTaken: r.data.taken }));
  }, 300);
}

Single-page app submissions with Axios

SPAs don't use form POSTs with page reloads. Instead they use JavaScript HTTP requests. Axios is the most common choice — promise-based, works in both browser and Node.

GET (fetching) and POST (submitting)
// Fetching data
axios.get('/api/users')
  .then((response) => {
    // response.data — JSON-parsed body
    // response.status — e.g. 200
  })
  .catch((err) => {
    // err.response.status — e.g. 404, 500
    // !err.response → network error
  });

// Uploading data
axios.post('/api/comments', { text: 'Great photo!' })
  .then((response) => { /* success */ })
  .catch((err) => { /* error */ });

Server-side validation with Mongoose

Your database schema is the last line of defense. Mongoose allows you to attach validator functions to any field:

const userSchema = new Schema({
  phone: {
    type: String,
    validate: {
      validator: (v) => /\d{3}-\d{3}-\d{4}/.test(v),
      message: '{VALUE} is not a valid phone number!'
    }
  }
});

For complex integrity rules — like "only the author can delete their comment" or "free users are limited to 50 photos" — you need custom application logic, not just schema validators.

A user fills in a registration form and your React component validates the email format client-side. Is server-side validation still needed?
Correct! Client-side validation is purely a UX feature — it gives users fast feedback. But any developer can use curl, Postman, or a custom script to send raw HTTP requests directly to your API, bypassing every browser check. Server-side validation is non-negotiable.
