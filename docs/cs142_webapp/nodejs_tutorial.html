01
What is Node.js? The JavaScript runtime that brought JS to the server

Node.js takes Chrome's V8 JavaScript engine — the one powering your browser — and runs it on the server. The result: the same JavaScript language, everywhere.

  • Takes a JavaScript engine from a browser (Chrome's V8) — same JS on both browser and server, no DOM needed on the server
  • Includes DOM-like events and an event queue — everything runs as a call from the event loop
  • Makes event interface to all OS operations — wraps all OS blocking calls (file and socket/network I/O)
  • Adds a proper module system (predated import/export) — each module gets its own scope

02
Threads vs Events Two fundamentally different models for handling concurrency

Traditional servers use threads — each request gets a thread that blocks while waiting for I/O. Node.js uses events — a single thread handles many requests by never blocking.

🧵 Thread-Based (Blocking)
request = readRequest(socket); reply = processRequest(request); sendReply(socket, reply); // Thread switches + scheduler // Each request = 1 thread
⚡ Event-Based (Non-blocking)
startRequest(socket); listen("requestAvail", processRequest); listen("processDone", sendReply); // Event queue processing // 1 thread handles many requests

With callbacks, the event-based approach looks like this in JavaScript:

callback-style.js
readRequest(socket, function(request) { processRequest(request, function(reply) { sendReply(socket, reply); }); });

The Event Queue The engine that drives all of Node.js

At the core of Node.js is a simple but powerful loop. It continuously checks for events to process, and dispatches them one at a time.

Node Modules The module system that predated ES6 imports
require-examples.js
// System module — looks in node_modules directories require("fs"); // From a file — reads specified file require("./XXX.js"); // From a directory — reads myModule/index.js require("./myModule");

Module files have a private scope — variables declared inside are not global. You explicitly export what you want to share:

myModule.js
var notGlobal; // Private to this file function func1() {} function func2() {} // Export the public interface module.exports = { func1: func1, func2: func2 };
💡
Can also use ES6 import/export syntax if your file is named *.mjs — but require() remains the traditional Node.js way.
05
The Buffer Class Efficient binary data handling outside the V8 heap

Manipulating binary data isn't JavaScript's strength — but web servers do it constantly: DBMS ⇔ Web Server ⇔ Browser.

Node's Buffer class solves this by allocating memory outside the V8 heap, with an interface that looks like an array of bytes.

  • Used by all wrapped OS I/O calls (fs, net, …)
  • Optimized sharing with pointers rather than copying (buffer.copy())
  • For example: efficient pipe from fs.readFile to socket.write
buffer-operations.js
// Convert Buffer to string — pick your encoding buf.toString("utf8"); // UTF8 — most common on the web buf.toString("hex"); // Hex encoding (2 digits per byte) buf.toString("base64"); // Base64 encoding // Reading a file returns a Buffer by default let fs = require("fs"); fs.readFile("smallFile", function(error, dataBuffer) { if (!error) { console.log(dataBuffer.toString()); // Converts Buffer → String } });
06
Programming with Callbacks The mental model for async code

The key insight: if your code doesn't block, it's the same as thread programming. If it does block (or needs to wait), you need a callback. Often what was a return statement becomes a function call.

🧵 Threads — Sequential
r1 = step1(); console.log('step1 done', r1); r2 = step2(r1); console.log('step2 done', r2); r3 = step3(r2); console.log('step3 done', r3); console.log('All Done!');
⚡ Callbacks — Nested (Correct)
step1(function(r1) { console.log('step1 done', r1); step2(r1, function(r2) { console.log('step2 done', r2); step3(r2, function(r3) { console.log('step3 done', r3); console.log('All Done!'); // ✅ Here! }); }); });
⚠️
Common mistake: Putting console.log('All Done!') after the callback chain — it runs immediately, before any async work finishes. Always place completion logic inside the innermost callback.
07
EventEmitter Pattern Listen for events, emit signals — the backbone of Node.js

The listener/emitter pattern is the foundation of event-driven programming. A listener is a function that runs when an event fires. An emitter signals that an event occurred.

event-emitter.js
const EventEmitter = require('events'); const myEmitter = new EventEmitter(); // Register a listener with .on() myEmitter.on('myEvent', function(param1, param2) { console.log('myEvent occurred with ' + param1 + ' and ' + param2 + '!'); }); // Signal the event with .emit() myEmitter.emit('myEvent', 'arg1', 'arg2'); // → "myEvent occurred with arg1 and arg2!" // Multiple event types — typical pattern myEmitter.on('conditionA', doConditionA); myEmitter.on('conditionB', doConditionB); myEmitter.on('error', handleError); // Always handle errors!
💀
Always handle the 'error' event! If an error event is emitted with no listener registered, Node.js will throw an exception and exit the process.

TCP Networking Building network servers with Node's net module

Node's net module wraps the OS's network routines, including higher-level helpers like createServer.

tcp-server.js
let net = require('net'); // Creates socket, binds port 4000, listens for connections // Calls processTCPconnection on each new TCP connection net.createServer(processTCPconnection).listen(4000); function processTCPconnection(socket) { socket.on('data', function(data) { // Handle incoming data }); socket.on('end', function() { // Handle disconnect }); }
10
Real Examples A chat server, a file server, and the N-files problem

Example 1: Chat Server — broadcast messages to all connected clients.

chat-server.js
let clients = []; function processTCPconnection(socket) { clients.push(socket); socket.on('data', function(data) { broadcast("> " + data, socket); }); socket.on('end', function() { clients.splice(clients.indexOf(socket), 1); }); } function broadcast(message, sender) { clients.forEach(function(client) { if (client === sender) return; client.write(message); }); }

Example 2: Simple File Server — receive a filename, send back the file contents.

file-server.js
net.createServer(function(socket) { socket.on('data', function(fileName) { fs.readFile(fileName.toString(), function(error, fileData) { if (!error) { socket.write(fileData); // Writing a Buffer } else { socket.write(error.message); // Writing a String } socket.end(); }); }); }).listen(4000);
🔍
Think about the concurrency here: multiple clients can request files simultaneously, and Node handles them all in a single thread — because fs.readFile is non-blocking.

Example 3: Reading N Files in Parallel — use the async module to handle completion.

parallel-reads.js
var async = require('async'); var fileContents = {}; async.each(['f1', 'f2', 'f3'], readIt, function(err) { if (!err) console.log('Done'); // fileContents is fully filled in if (err) console.error('Error:', err.message); }); function readIt(fileName, callback) { fs.readFile(fileName, function(error, dataBuffer) { fileContents[fileName] = dataBuffer; callback(error); }); }
