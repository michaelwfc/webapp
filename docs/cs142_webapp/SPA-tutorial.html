01

What is a Single Page Application?

A Single Page Application (SPA) is a web app that loads one HTML document and dynamically updates content in the browser without performing a full page reload. The JavaScript runtime persists across views — giving users the experience of a fast, native-like app.

Unlike traditional multi-page apps that fetch a new HTML page from the server on every navigation, SPAs intercept navigation events, update only what's needed, and manage history entirely on the client.

Traditional MPA

Every click triggers a full HTTP request. Server sends back a new HTML page. JS environment is torn down and restarted. State is lost.

Single Page App

One initial load. All subsequent navigation is handled in JS. DOM updates dynamically. State, network connections, and timers persist.

Key insight: The "single page" refers to the server's perspective — it serves one HTML document. From the user's point of view, the app has many screens.

02

Web Apps & Browser Expectations

Web apps run in browsers, and users have deeply ingrained expectations about how browsers behave. Understanding these is the foundation of SPA design.

URL History

Browsers maintain a visited-URL history. The back button navigates to the previous URL, and the forward button moves ahead.

Page Navigation

Users navigate by typing in the location bar, selecting bookmarks, clicking links, or hitting the refresh button.

JS Environment

When the browser navigates to a new URL, it discards the current JavaScript environment. Only the URL and cookies survive.

The core problem: JS frameworks build state in memory. Browser navigation destroys that state. SPAs need to bridge this gap — mapping application state to the URL so state can survive navigation.
03

The Problem with Early Web Apps

Early web apps evolved through two phases, both with painful trade-offs for users and developers.

Phase 1 — Server-Rendered Multi-Page

Pre-JS era

Each page had a distinct URL. Navigation fetched a new HTML page from the server. Clean URL story, but zero shared JS state — every click was a cold start.

Phase 2 — Single URL JavaScript Apps

Early JS era

Website became a single URL with JavaScript rendering everything. No more cold starts — but navigating away or refreshing destroyed all user state.

The dreaded "All will be lost!" warning on page unload.
JavaScript — the panic guard 
// Warn users before accidentally navigating away
window.onbeforeunload = function(e) {
  return 'All will be lost!';
};

Users expect their apps to behave correctly in the browser. A proper SPA must support all of these:

Navigate with forward and back buttons
Navigate away and return to the same state
Bookmark a specific place in the app
Copy the URL and share it with someone
Push the page refresh button safely

04

Changing the URL Without Reloading

The key technique behind SPAs is updating the browser's URL without triggering a page reload. There are two mechanisms for this.

Mechanism 1 — Hash Fragments

The hash portion of a URL (#fragment) can be changed freely without a reload. The browser doesn't send it to the server — it's purely client-side.

http://example.com
http://example.com#/profile
http://example.com?id=3535
http://example.com?id=3535#/settings

Mechanism 2 — HTML5 History API

HTML5 gave JavaScript full control over the browser's history stack via window.history. This allows changing the URL path without sending any request to the server.

JavaScript — History API 
// Push a new URL into history (no reload!)
window.history.pushState(
  { page: 'profile' },  // state object
  'Profile',           // title (ignored by most browsers)
  '/user/profile'       // new URL path
);

// Replace current history entry
window.history.replaceState(state, title, url);

// Listen for back/forward navigation
window.addEventListener('popstate', (e) => {
  console.log(e.state); // restore app state from URL
});
React Router uses both: HashRouter uses the hash fragment approach (works without server config). BrowserRouter uses the HTML5 History API (requires server to serve the app for all routes).
05

Deep Linking

Deep linking means the URL encodes enough context that directing a browser to that URL will restore the app to the exact same state. This enables bookmarking, sharing, and refreshing.

Core concept: "The URL should capture the web app's context so that directing the browser to the URL will result in the app's execution resuming at that context."

There are two approaches to implementing deep linking:

Approach 1 — Context State in the URL

Automatic
Works for browser back/forward navigation and refresh
User can copy URL from the location bar at any time
No explicit user action needed — every state is linkable

Approach 2 — Share Button Generates URL

On-demand
App only generates a shareable URL when user explicitly requests it
Keeps the location bar clean and human-readable
Allows separating "shareable" state from ephemeral navigation
Either way: the app must be able to initialize itself from a deep-linked URL. When a user arrives at /Book/Moby/ch/3, the app must parse that URL and render the correct content — even on a cold start.

Ugly URLs — a UX concern

Deep linking doesn't have to mean ugly URLs. Compare:

http://www.example.org/dirmod?sid=789AB8&type=gen&mod=CorePages&gid=A6CD4967199
versus
http://www.example.org/show/A6CD4967199

Use clean, semantic URL patterns. They're shareable, memorable, and communicate intent. This is one reason the BrowserRouter approach is preferred when server configuration allows it.

ReactJS & React Router

React has no built-in opinion on routing — it's a view library. For SPA routing, you need a third-party package. The most popular choice is React Router.

Terminal — install React Router

JavaScript — import React Router V5 
import { HashRouter, Route, Link, Redirect }
  from 'react-router-dom';

    /* Navigation Links */
    <Link to="/states">States</Link>
    <Link to="/profile">Profile</Link>

    /* Route declarations */
    <Route path="/states"  component={States}  />

HashRouter

Uses the URL hash (#/path). No server config required. Hash changes are never sent to the server.

BrowserRouter

Uses the HTML5 History API. Clean URLs (/path). Requires server to serve index.html for all routes.

07

Passing Parameters via Routes

Routes can contain dynamic segments — parts of the URL path that act as parameters. React Router captures these and passes them to the component as props.match.params.

JSX — Route with URL parameters 
// Define the route — :book and :chapter are params
<Route
  path="/Book/:book/ch/:chapter"
  component={BookChapterComponent}
/>

// Link that populates the params
<Link to="/Book/Moby/ch/1">Moby Dick, Ch 1</Link>
// Renders: <a href="#/Book/Moby/ch/1">...</a>
JSX — Component receiving route params 
// Destructure match from props
function BookChapterComponent({ match }) {
  return (
    <div>
      <h3>Book: {match.params.book}</h3>
      <h3>Chapter: {match.params.chapter}</h3>
    </div>
  );
}

Given the URL /Book/Moby/ch/1, the match.params object will be:

JSON — match.params 
{
  "book":    "Moby",
  "chapter": "1"
}
Tip: URL params are always strings. If you need a number, parse with parseInt(match.params.chapter) or the unary + operator.
08

Route Props: component, render, children

React Router's <Route> supports three rendering props with different semantics. Understanding the difference is key to getting the right behaviour.

Pass a React component class or function. React Router will instantiate it when the path matches.

<Route path="/book" component={BookComponent} />
  • Mounts the component when URL matches, unmounts when it doesn't
  • Passes a match object with params, url, and history
  • Use when you want a fresh component instance on each route match

Pass an inline render function. Useful for passing extra props without causing unnecessary remounts.

<Route
  path="/book"
  render={props => (
    <BookComponent
      book={props.match.params.book}
      onFavorite={handleFavorite}
    />
  )}
/>
  • Calls the function with route props (including match) when URL matches
  • Does not unmount/remount the component on URL change — updates instead
  • Preferred when you need to pass extra props alongside route params

Like render= but the function is called regardless of whether the URL matches.

<Route
  path="/book"
  children={({ match }) => (
    <li className={match ? 'active' : ''}>
      <Link to="/book">Books</Link>
    </li>
  )}
/>
  • match is null when the URL doesn't match
  • Useful for nav items that are always rendered but styled differently when active
  • Always runs, even when the path doesn't match
Precedence order: When multiple render props are present, React Router uses: component > render > children. Use <Switch> to render only the first matching route.
JSX — Switch renders first match only 
<Switch>
  <Route exact path="/"       component={Home}    />
  <Route path="/book/:id"  component={BookDetail} />
  <Route path="/book"     component={BookList}  />
  <Route                   component={NotFound}  />
</Switch>

Designing URL State

Not everything belongs in the URL. Choosing what to encode in the URL is a design decision that affects shareability, bookmarkability, and user expectations.

The Decision Framework

UI State In URL? Rationale
Current page / view✓ YesCore navigational state — always shareable
Selected item (e.g. book ID)✓ YesUser expects this to survive a refresh
Search query✓ YesShareable search results are expected
Sort column & direction~ MaybeDepends on use case — share vs. personal pref
Table page / viewport offset~ MaybeUseful for sharing but can clutter the URL
Open modal / dialog✗ NoEphemeral interaction — not a shareable context
Hover / focus state✗ NoTransient UI state — meaningless in a URL
Unsaved form values✗ NoToo volatile; use local state instead
Rule of thumb: Put in the URL what you'd want to restore on a browser refresh or share with someone. Leave out what is purely ephemeral UI interaction state.

Example: Data Table URL Design

Consider a sortable, paginated data table. What state should live in the URL? Think through the user scenarios:

Bookmarking for Self

Sort column + direction and current page could be valuable — you'd want to return to exactly where you were.

Sharing with Others

Search query and filter state are almost certainly shareable. But your current scroll position is not.

Nav Away & Back

The browser back button should restore the table view — consider using the History API's state object for this.
