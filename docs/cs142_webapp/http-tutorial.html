HTTP is a simple request-response protocol layered on top of TCP/IP. The browser opens a connection, sends a request, and reads back the response — that's the entire cycle.

02 — Addressing

Universal Resource
Locators

Every resource on the web has a URL. Understanding each part is essential for building and debugging web applications.

http

All three steps happen every time you load a URL:

1. Establish a TCP/IP connection to www.example.com:80
2. Send an HTTP GET request along the connection
3. Read the server's response from the connection
Omitting the port defaults to :80 for http and :443 for https. Omitting the path defaults to /index.html (server-configured).
03 — Client → Server

Anatomy of an
HTTP Request

A request is written as plain text lines to a socket. The format is strict: method line, then headers, then a blank line, then an optional body. 

REQUESTGET /index.html HTTP/1.1         ← Method  Path  Version
Host:             www.example.com
User-Agent:       Mozilla/5.0
Accept:           text/html, */*
Accept-Language:  en-us
Accept-Charset:   ISO-8859-1,utf-8
Connection:       keep-alive
                                        ← blank line (required)
                                        ← body (optional, used by POST/PUT)

The three parts

Request line — method, resource path, and HTTP version. Always first.

Headers — key-value pairs providing metadata: who's asking, what they accept, connection preferences.

Body — optional payload. Used when submitting form data (POST) or uploading content (PUT). Absent in GET requests.

04 — Verbs

HTTP Methods

Each method signals a different kind of intent. REST APIs use all five; traditional web pages rely mostly on GET and POST.

GET
Fetch a resource. No body. Safe and idempotent — repeated calls have no side effects.
READ
POST
Send form data to a URL and receive a response. Has a body. Not idempotent.
CREATE
PUT
Store or replace a resource at a URL. Idempotent — same request gives same result.
UPDATE
DELETE
Remove a resource at the given URL. Idempotent by convention.
DELETE
HEAD
Same as GET but the server only returns headers — no body. Useful for checking if a resource exists.
INSPECT
Traditional web forms only support GET and POST. REST APIs leverage all four data methods (GET, POST, PUT, DELETE) to map neatly onto CRUD operations.
05 — Server → Client

Anatomy of an
HTTP Response

The server replies with a status line, headers, a blank line, and then the body — the actual resource content. 

RESPONSEHTTP/1.1 200 OK                      ← Version  Code  Message
Date:           Thu, 24 Jul 2008 17:36:27 GMT
Server:         Apache-Coyote/1.1
Content-Type:   text/html;charset=UTF-8
Content-Length: 1846
                                           ← blank line
<?xml ... >
<!DOCTYPE html ... >
<html ...>
  ...
</html>                                   ← HTML body

Status line breakdown

HTTP/1.1 — the protocol version.

200 — numeric status code that the browser acts on programmatically.

OK — human-readable message accompanying the code.

07 — Performance

Browser Caching
Control

Servers can instruct browsers to cache responses using the Cache-Control header. The browser may reuse a cached response until it exceeds the max-age. 

HTTP Response Header:

Cache-Control: max-age=120     ← expires in 2 minutes
Cache-Control: max-age=86400   ← expires in 1 day
Cache-Control: no-cache        ← always revalidate
Cache-Control: no-store        ← never cache (sensitive data)

Most useful on static assets: images, CSS stylesheets, compiled JavaScript bundles.

Benefits

  • Reduces app startup latency significantly
  • Lowers server load and bandwidth costs
  • Enables offline-like experience for static content

Trade-offs

  • Changes may not be picked up immediately by users
  • Stale assets after a new deployment can cause bugs
  • Requires cache-busting strategy (content hashing)
Best practice: use long max-age (e.g. 1 year) for versioned static files (e.g. app.a3f2c1.js) and short or no-cache for HTML files that reference them.
08 — Browser Lifecycle

How the Browser
Fetches Resources

A browser's entire job is fetching things via HTTP. Not all fetches are created equal — they happen in three different modes depending on how the resource is referenced.

01

Synchronous (Blocking)

Page rendering pauses until the response arrives. Used for critical CSS and scripts. 

<link href="styles.css"
      rel="stylesheet" />
<script src="app.js"></script>
02

Asynchronous & Parallel

Multiple resources fetched simultaneously without blocking the page. Common for images. 

<img src="hero.jpg">
<img src="thumb1.jpg">
<img src="thumb2.jpg">
03

Background (Prefetch)

Linked URLs may be prefetched quietly in the background by the browser. 

<a href="http://example.com">
  Link
</a>

A very common error you'll see in the browser console:

GET http://localhost:3000/favicon.ico 404 (Not Found)

This means the browser attempted a fetch, but the server had no matching resource to return.

09 — Advanced

Dynamic Script
Loading

JavaScript can trigger HTTP requests at runtime by injecting HTML into the page. This is powerful — but also a significant security surface to be aware of. 

// This uses HTTP to fetch myJS.js and runs it immediately
elm.innerHTML =
  '<script src="http://www.example.com/myJS.js"' +
  'type="text/javascript"></script>';

When the browser parses this injected HTML, it immediately makes an HTTP GET request for myJS.js and executes it in the current page context.

This pattern is useful for lazy-loading scripts on demand. However, loading scripts from untrusted third-party URLs is a critical XSS risk — only load scripts from sources you fully control or trust.
