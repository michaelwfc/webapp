Sessions & Cookies

HTTP is a protocol with no memory — every request arrives at the server as if it's the first time the two have ever met. Yet websites remember you, keep you logged in, and maintain your preferences across dozens of page loads. How? This tutorial traces the full story, from the fundamental statelessness problem all the way to modern Express session middleware and the Web Storage API.

Section 01

The Core Problem: Recognizing Who's Talking

Picture a bank teller with a very unusual condition: they have no short-term memory. Every single customer who walks up to the counter is a complete stranger. To do anything useful — check an account balance, process a transfer, approve a withdrawal — the teller needs to figure out who this person is before they can do anything else.

Web servers face exactly this challenge. When an HTTP request arrives, it carries almost no inherent identity. The server knows the request came from some IP address, but IP addresses are shared, dynamic, and easy to spoof. There's nothing in the raw request that reliably says "this is Alice" or "this request is a continuation of the session that started ten minutes ago."

And yet every useful web application needs exactly this. When you're browsing a shopping site, the server needs to know whose cart you're looking at. When you click "Checkout," the server needs to be sure you're the same person who added those items. When you load your inbox, the server needs to know which emails belong to you — not to the next person who happens to send a request from a shared WiFi network.

This need is called session state — a persistent record of who you are and what you've been doing, that travels with you across every request in a browsing session. Without it, every request is anonymous, and every interaction starts from scratch.

Every route handler faces the same question: who is making this request?
expressApp.get('/user/:user_id', function (httpRequest, httpResponse) {
  // Before doing anything useful, we need to know who this is.
  // Can we trust this request? Is the user authenticated?
  let sessionState = GetSessionState(httpRequest);

  // Only then can we decide: serve the data, or reject the request.
});

This pattern repeats in every single route handler across the entire app. Session state is the foundation everything else is built on — get it wrong, and your app either leaks data to the wrong people or rejects legitimate users.

Key Takeaway

Web applications need to know who is making each request and maintain that identity across many separate HTTP requests. This is called session state — and solving the problem of how to create, store, and securely transmit it is what this entire tutorial is about.

Section 02

Why HTTP Has No Memory: Statelessness Explained

To understand why session state is hard, you first need to understand a foundational property of HTTP: it is stateless. This isn't a bug — it was a deliberate design decision. And once you understand why, the solutions start to make intuitive sense.

When the web was being designed in the early 1990s, the goal was simplicity and scalability. HTTP was designed as a request-response protocol where each request is completely independent of every other request. The server receives a request, does its work, sends back a response, and then forgets everything. There is no persistent connection, no memory of previous interactions, no concept of a "conversation."

This design choice made the web incredibly scalable: any server in a cluster can handle any request because no server needs to remember anything about previous requests. You can add more servers, take servers offline, and route requests however you like. The protocol simply doesn't care.

The tradeoff is that the protocol has no concept of identity. Look at where session state could live in the system, and you'll see the problem immediately:

Where does session state live in a web architecture?
Web Browser ←——— HTTP ———→ Web Server ←——— LAN ———→ Storage System
Maybe here? ❌ Stateless — can't store anything here ✓ Here — persistent storage

The web server sits in the middle but cannot hold state — it processes a request and moves on. Session data must be stored either in the browser or in a persistent storage system, with a reference passed through HTTP so the server can find it.

The web server is the one place you can't reliably store session data. It processes each request in isolation and may be one of many servers behind a load balancer. The browser and the storage system are the two candidates — and the challenge is deciding what to put where, and how to pass the reference between them securely.

Key Takeaway

HTTP is stateless by design — each request is independent, and the web server remembers nothing between them. Session data must live in the browser or in a backend storage system. The web server's job is to look up the right session using a reference that arrives with each request.

Section 03

The Session Lookup Problem: Connecting Requests to Identity

So session data must live outside the web server. But that immediately raises a practical question: if each HTTP request is anonymous, how does the server know which session it belongs to?

The answer is that something must be included in the request itself — a token, an identifier, a reference — that tells the server "this request belongs to session #12345." The server receives this token, looks up the associated session data, and uses it to authenticate and personalize the response.

This sounds simple, but there's an important constraint: the system must be resistant to forgery. If session identifiers are easy to guess or manufacture, then a malicious user could craft a request that pretends to belong to someone else's session. Imagine guessing a session ID and suddenly seeing — or acting as — a different user. This is called session hijacking, and it's a real attack that has affected many systems.

Over the history of the web, two main mechanisms have emerged for carrying this session reference:

Cookies were the original solution, introduced in the early days of the commercial web. The server tells the browser to store a small piece of data, and the browser dutifully includes that data in every subsequent request to the same server. Cookies became the standard way to maintain session state for decades.

The Web Storage API (localStorage and sessionStorage) is a more modern alternative. Unlike cookies, Web Storage data isn't automatically sent to the server — you have to explicitly include it in requests. This gives developers more control, but also more responsibility.

Key Takeaway

Every request needs to carry a session token that tells the server which session it belongs to. This token must be hard to guess or forge — otherwise, an attacker could impersonate another user. Cookies and the Web Storage API are the two standard mechanisms for carrying this token in the browser.

Section 04

How HTTP Cookies Work

A cookie is a small piece of data — a name-value pair — that the server plants in the browser during one response, and then receives back automatically in every subsequent request to the same domain. Think of it like a coat check ticket: the venue (server) gives you a numbered stub (cookie) when you arrive, and you hand it back every time you want to retrieve something from the coat room.

The mechanics work through two HTTP headers. When the server wants to set a cookie, it includes Set-Cookie headers in its response. The browser stores these cookies locally and automatically includes them in a Cookie header on every future request to that domain.

Server sets cookies in the HTTP response headers
// The server adds these to its HTTP response
Set-Cookie: cookie_name1=cookie_value1
Set-Cookie: cookie_name2=cookie_value2; expires=Sun, 16 Jul 2016 06:23:41 GMT

Each Set-Cookie header creates one cookie. The browser stores both and will include them in future requests. The optional expires attribute tells the browser when to automatically delete the cookie.

Browser sends cookies back automatically on every subsequent request
// The browser automatically adds this header to every request
Cookie: cookie_name1=cookie_value1; cookie_name2=cookie_value2

The browser bundles all matching cookies into a single Cookie header. The server reads this header and knows exactly which session to look up — without the user doing anything at all.

The word "automatically" is key here. The developer doesn't write JavaScript to send cookies — the browser handles that entirely. This is both the power of cookies (zero developer effort to transmit) and a source of their security concerns (the browser sends them everywhere, including to malicious pages that trigger requests to your domain).

Key Takeaway

Cookies work through a two-step handshake: the server sets them with Set-Cookie response headers, and the browser sends them back with every request via the Cookie header — automatically, without any JavaScript required. This automation is what makes cookies so effective for session management, and what also makes them a frequent security consideration.

Section 05

What Lives Inside a Cookie

A cookie is more than just a key-value pair — it comes with a set of attributes that control when, where, and to whom it's sent. Understanding these attributes is essential for writing secure, well-behaved web applications.

Name & Data

The payload

The actual key-value pair. The name identifies the cookie; the value is the data. For session cookies, the value is typically an opaque session ID — not readable data.

Domain

Scope restriction

Specifies which server (and optionally which port and URL prefix) the cookie is sent to. The browser only includes a cookie in requests that match its domain — cookies from your bank don't go to your shopping site.

Expiration

Lifetime control

An optional date after which the browser automatically deletes the cookie. Without an expiration, the cookie is a "session cookie" — it lives only as long as the browser tab is open.

Two hard limits constrain how cookies can be used as storage. First, cookies are tiny: browsers typically enforce a maximum size of 4 KB per cookie. That's enough for an ID or a small amount of metadata, but far too small for anything substantial. Second, browsers limit the number of cookies per domain — typically around 50 cookies. These limits exist to prevent abuse and to keep HTTP headers from ballooning in size.

These constraints are a hint about the intended use. Cookies were designed to carry session references, not session data. The right pattern is to store a small, opaque identifier in the cookie, and keep all the actual data on the server where there are no such size limits.

Key Takeaway

Cookies carry a name-value payload plus attributes for domain scoping and expiration. They're tightly constrained in size (4 KB max) and count (~50 per domain). These limits aren't limitations to work around — they're a signal that cookies are intended to carry references to session data, not the data itself.

Section 06

The Reliability Problem: Why Cookies Make Poor Storage

If you've ever built something that relies on a cookie being there when you need it, you may have learned the hard way: cookies are not reliable storage. They are exposed to the user and the environment in ways that most storage systems aren't.

Consider everything that can go wrong with a cookie. The user opens their browser settings and deletes all cookies — gone. They switch from Chrome to Safari — gone. They install a privacy extension that blocks or clears cookies — gone. A cross-site scripting (XSS) vulnerability in your app lets a malicious script read the cookie — stolen. A man-in-the-middle attack on an unencrypted connection intercepts the cookie — stolen.

Even without malice, cookies have a troubled history. For years, they were used for pervasive tracking across websites — following users from page to page, site to site, building profiles without consent. This history means that privacy-conscious users often block or clear cookies as a matter of routine. Users are suspicious of them for good reason.

Important Constraint

Because users can view, modify, corrupt, delete, or steal cookies at any time, they should only be used for data that can be safely lost or recovered — hints, preferences, and session tokens. Never store sensitive information (passwords, personal data, authorization tokens) directly in a cookie value without cryptographic protection.

The practical upshot is that cookies are best suited for one specific purpose: carrying a session token back and forth so the server can look up the real session data in a secure backend store. Everything else — the actual user data, preferences, permissions — lives server-side where the user can't touch it.

Key Takeaway

Cookies are exposed and fragile: users can delete, modify, or lose them, and attackers can steal them. Design your app to tolerate missing cookies gracefully, and never put sensitive data in a cookie value. Use cookies for what they're good at: carrying an opaque session token back to the server.

Section 07

Session Cookies in Practice: How Rails Solved It First

Early web frameworks had to invent session management from scratch. Ruby on Rails set the pattern that most modern frameworks — including Express — have adopted in some form. Understanding how Rails approached it makes the Express solution feel familiar rather than arbitrary.

Rails gave developers a session object that behaved like a simple JavaScript object — you could read and write arbitrary data to it, and Rails handled all the HTTP plumbing transparently:

Rails session — write to it like a hash; Rails handles the rest
# Store the authenticated user's ID in the session
session[:user_id] = "mendel"

One line of Ruby. Rails took care of packaging this into a cookie, signing it cryptographically, sending it with the response, reading it back on the next request, and making it available in the controller. The developer never touched an HTTP header directly.

Rails followed a consistent lifecycle for every request, which is the same lifecycle modern frameworks use today:

Check for a session cookie

At the very start of each request, before any route handler runs, the framework looks for a session cookie in the request headers.

Load or create the session

If a valid session cookie is found, the framework loads the associated session data. If not — or if the cookie is invalid — it creates a brand-new empty session.

Run your route handler

Your application code runs with full access to the session object — read existing data, write new data, or clear it on logout.

Save and send the cookie

At the end of the request, the framework saves any session changes and ensures the session cookie is included in the response — creating one if it's a new session.

Key Takeaway

Session middleware follows a consistent four-step lifecycle on every request: find the session cookie → load the session → run your handler → save and send the cookie. Your code only sees the session object — the framework handles all the HTTP mechanics transparently. This is the same pattern Rails pioneered and Express adopted.

Section 08

Session Security: Never Trust Cookie Data

Because cookies live in the browser — a place the user controls — a critical security principle applies: never store sensitive data directly in a cookie value without protection. The data can be read, and it can be modified.

Consider what happens if an early developer naively stores session state directly in a cookie:

The dangerous anti-pattern — sensitive data stored naked in a cookie
// ❌ Never do this — both values are visible in the browser
session.user_id = "mendel";
session.password = "foobar";  // Catastrophically dangerous!

Anyone who opens their browser's developer tools can read these values in plain text. An attacker who steals the cookie (via XSS, network interception, or physical access) gets the password immediately. This is a foundational security mistake.

Cryptography provides two tools for making cookies safer. Encryption hides the content of the cookie from anyone who reads it — even if a cookie is stolen, the thief can't read the data without the server's secret key. Digital signing (often called a MAC or HMAC) doesn't hide the data, but detects if it's been tampered with — if the user modifies a signed cookie, the server knows and rejects it.

But even with cryptography, there's a cleaner architectural choice: don't put session data in the cookie at all. Store only an opaque session ID — a random, unforgeable token that means nothing on its own — and keep all the actual data on the server. The cookie is just a reference:

The right approach — an opaque session ID that means nothing without the server
// ✓ Correct — the cookie value is an opaque reference, not real data
Set-Cookie: session=0x4137fd6a; Expires=Wed, 09 Jun 2012 10:18:14 GMT

0x4137fd6a is just a random-looking token. Even if someone steals it, they can't decode your user ID, permissions, or any other session data from it — that all lives on the server. The token still needs to be signed to prevent forgery.

This approach offers three security benefits: less data transmitted on every request, nothing sensitive exposed in the browser, and a session ID that can be invalidated server-side (on logout) even if the token isn't expired yet.

Key Takeaway

Never store real data in cookies without cryptographic protection. The gold standard is to store only an opaque session ID in the cookie and keep all actual session data on the server. The session ID must be cryptographically signed so the server can detect tampering — and it can be invalidated on logout even before it expires.

Section 09

Where to Store Session Data: Three Options

Once you've decided to keep session data on the server rather than in the cookie, you need to choose a storage backend. There are three main options, each with a distinct performance and complexity profile. The right choice depends on your scale, infrastructure, and how long sessions need to live.

Storage location Speed Advantages Disadvantages
Web server memory fastest Zero network latency — data is right there in the process. Simplest to set up for development. Sessions are lost when the server restarts. Doesn't work with multiple servers (load balancers route to different machines). Memory grows with active user count.
Database (e.g. MongoDB, PostgreSQL) medium Shared across all servers. Survives server restarts. Works naturally with your existing data infrastructure. A database round-trip on every single request adds latency. Databases are built for durability, not for small, frequently-read transient data — this may be overkill.
Specialized in-memory store (Redis, Memcache) fast Designed exactly for this use case: fast, shared, short-lived data. Shared across all servers. Supports automatic expiration. Scales well. Another piece of infrastructure to deploy, monitor, and maintain. Adds operational complexity to your stack.

For most production applications, Redis is the standard choice for session storage. It's fast (data lives in memory), supports TTL-based expiration (sessions auto-delete after they expire), can be shared across many server instances, and has excellent client libraries for Node.js. Memcache is a similar option with a simpler feature set.

For development and small single-server deployments, in-process memory is fine. Just remember that every time you restart your dev server, all sessions are cleared — which means you'll be logged out.

Key Takeaway

In-process memory is great for development but breaks in production — it doesn't survive server restarts and doesn't scale across multiple servers. A database works but adds latency. Redis or Memcache are the production-standard choices: fast, shared, scalable, and purpose-built for the short-lived, frequently-accessed nature of session data.

Section 10

Express Session Middleware: From Concept to Code

Express doesn't include session management in its core, but the express-session middleware package provides everything you need — and follows the same lifecycle we described in the Rails section. Setting it up takes just a few lines of code.

Installing and wiring up express-session
const session = require('express-session');

// Mount the session middleware — must come before your route handlers
app.use(session({
  secret: 'yourSecretKey'  // used to sign the session ID cookie cryptographically
}));

// Now every route handler gets httpRequest.session for free
app.get('/profile', function (httpRequest, httpResponse) {
  // httpRequest.session is an object — read it, write it, it persists
});

The secret option is how express-session signs the session ID cookie to prevent forgery. In production, this should be a long, random string stored in an environment variable — never hardcoded in source code.

Reading, writing, and destroying sessions

Once the middleware is mounted, working with sessions in your route handlers is straightforward. The httpRequest.session object behaves like a plain JavaScript object — you can add properties, read them, and delete them. Changes are automatically persisted to the session store at the end of each request.

Full session lifecycle — login, use, and logout
// ── Login handler ─────────────────────────────────────
app.post('/login', function(req, res) {
  // After verifying credentials, store the user ID in the session
  req.session.user_id = "mendel";
  res.redirect('/');
});

// ── Protected route ───────────────────────────────────
app.get('/dashboard', function(req, res) {
  if (!req.session.user_id) {
    // No session — this user is not logged in
    return res.redirect('/login');
  }
  // Session exists — we know who this is
  res.render('dashboard', { userId: req.session.user_id });
});

// ── Logout handler ────────────────────────────────────
app.post('/logout', function(req, res) {
  // Destroy the session entirely — not just clear one field
  req.session.destroy(function(err) {
    res.redirect('/login');
  });
});

On logout, always call session.destroy() rather than just setting fields to null. This removes the session from the store entirely, invalidating the session ID even if someone has a copy of the cookie.

Switching to a production session store

By default, express-session stores sessions in Node.js process memory — which prints a warning in the console and is explicitly not recommended for production. The fix is to provide a store option pointing to a persistent backend. Here's how to hook it up to MongoDB via Mongoose:

Using MongoDB as the session store via connect-mongo
const MongoStore = require('connect-mongo')(express);

expressApp.use(session({
  secret: process.env.SESSION_SECRET,
  store: new MongoStore({
    mongooseConnection: mongoose.connection
  })
}));

connect-mongo uses the existing Mongoose database connection — no separate configuration needed. Sessions are stored as documents in MongoDB, survive server restarts, and can be shared across multiple server instances.

Key Takeaway

express-session gives you a req.session object in every handler — write to it on login, read from it to check authentication, and call session.destroy() on logout. For production, always configure a persistent store like Redis or MongoDB — the default in-memory store is development-only and loses all sessions on restart.

Section 11

The Modern Alternative: Web Storage API

Cookies have been around since 1994. In the decades since, browsers have added a more capable and developer-friendly storage mechanism: the Web Storage API. It comes in two flavors — sessionStorage and localStorage — and addresses many of cookies' ergonomic shortcomings.

sessionStorage

Per-tab, temporary storage

Data lives only while the browser tab is open. Closing the tab clears it entirely. Useful for wizard-style multi-step flows where you don't want data persisting after the user leaves.

localStorage

Persistent, cross-session storage

Data survives tab closures, browser restarts, and even reboots. Persists until explicitly cleared by JavaScript or the user. Useful for user preferences, cached data, and anything that should persist between visits.

Both APIs share the same simple key-value interface, making them easy to work with:

Web Storage API — three equivalent ways to read and write
// Writing — dot notation, setItem(), and bracket notation all work
localStorage.appSetting = 'dark-mode';
localStorage.setItem('appSetting', 'dark-mode');
sessionStorage['wizardStep'] = '3';

// Reading
const setting = localStorage.getItem('appSetting');
const step    = sessionStorage['wizardStep'];

// Removing a specific key
localStorage.removeItem('appSetting');

// Clearing everything
localStorage.clear();

Web Storage data stays in the browser — unlike cookies, it is never automatically sent to the server. If you want to include a token stored in localStorage in a request, you must read it with JavaScript and add it manually (typically as an HTTP header like Authorization: Bearer <token>).

Web Storage offers roughly 10 MB of space per origin — dramatically more than cookies' 4 KB. But the same reliability concerns apply: users can clear it through their browser settings, it's origin-scoped (so it doesn't travel to other domains), and it's accessible to any JavaScript running on the same page — meaning an XSS vulnerability can steal it just as easily as a cookie.

Key Takeaway

The Web Storage API (localStorage and sessionStorage) offers a clean key-value interface with far more capacity than cookies (~10 MB vs. 4 KB). The critical difference: Web Storage data is never sent to the server automatically — you must include it manually in requests. It shares cookies' reliability concerns and is equally vulnerable to XSS attacks.

Summary

Five Things to Remember

  • 1

    HTTP is stateless — you have to add identity yourself

    Every HTTP request is anonymous. The web server has no built-in memory of previous requests. Session state is the mechanism you layer on top to tie multiple requests together into a coherent, authenticated session — and building it correctly is the foundation of every web application that requires login.

  • 2

    Cookies are messengers, not storage

    The right role for a cookie is to carry an opaque session ID between the browser and the server — nothing more. All real session data should live server-side. Cookies are exposed to the user, size-limited to 4 KB, and vulnerable to theft; they're terrible general-purpose storage but excellent lightweight messengers.

  • 3

    Always sign session IDs cryptographically

    A session ID in a cookie is only as secure as its protection against forgery. Without cryptographic signing, any attacker can craft a cookie claiming any session ID and impersonate another user. express-session's secret option does this automatically — treat it like a password and never commit it to source control.

  • 4

    Use Redis or MongoDB for production session storage

    In-process memory is fine for development but fails in production: it doesn't survive restarts and doesn't scale across multiple servers. Redis is the standard production choice — fast, shared, and purpose-built for short-lived, frequently-accessed data. Always set a production store before going live.

  • 5

    Web Storage is powerful but not automatic

    localStorage and sessionStorage offer much more space than cookies and a cleaner API, but they share the same reliability concerns and are never sent to the server automatically. If you use Web Storage for authentication tokens, you're responsible for including them in every request you make — cookies handle this for free.
